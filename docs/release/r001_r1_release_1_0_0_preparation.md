---

## 1. Purpose

The purpose of this stage is to prepare the project for its first official public or freelance release.

Versions 1 to 8 were internal engineering milestones.

They represented the development journey of the project from prototype to freelance-ready delivery package.

Release 1.0.0 represents the first real release that can be shown to a client, included in a portfolio, or prepared for public presentation.

---

## 2. Naming Problem

The project has already completed Version 1 to Version 8 internally.

However, these versions should not be presented to the client as public product releases.

If the project is shown as Version 8 to a first client, it may create confusion.

The client may think the product already passed through eight public releases.

Therefore, the correct solution is to separate internal development versions from public release versions.

---

## 3. Internal Development Versions

The internal development cycle was:

- V1 — Working Prototype
- V2 — Polished Dashboard Prototype
- V3 — Data Source Abstraction
- V4 — CSV / Excel-Compatible Data Support
- V5 — Client Configuration
- V6 — Export Reports Foundation
- V7 — Unit Testing and Quality Gates
- V8 — Freelance Delivery Package

These versions are internal engineering milestones.

They can be mentioned in technical documentation, but they are not market releases.

---

## 4. Public Release Version

After completing V8, the project becomes ready for:

Release 1.0.0

This is the first official public or freelance release.

It should be used for:

- client delivery
- portfolio presentation
- public documentation
- freelance service packaging
- GitHub release if needed later

## 7. Release Versioning Rules

The release versioning system should follow semantic versioning style.

### Release 1.0.0

First official stable public/freelance release.

### Release 1.0.1

Small bug fixes or documentation corrections.

### Release 1.1.0

Small feature additions that do not change the core architecture.

### Release 2.0.0

Major architectural, commercial, or product-level change.

Examples of major changes:

- real file export
- native Excel support
- database integration
- authentication
- SaaS transformation
- major dashboard redesign

---

## 8. What Release 1.0.0 Includes

Release 1.0.0 includes the completed work from internal Versions 1 to 8:

- working Flutter dashboard
- polished dashboard UI
- data source abstraction
- CSV / Excel-compatible data support
- client configuration
- report preview foundation
- unit testing and quality gates
- freelance delivery documentation

---

## 9. What Release 1.0.0 Does Not Include

Release 1.0.0 does not include:

- real file saving
- PDF export
- Excel export
- external CSV file picker
- native Excel reading
- database integration
- authentication
- cloud deployment
- multi-user system
- full ERP system
- SaaS billing
- payment system

These should be considered future versions or separate paid phases.
